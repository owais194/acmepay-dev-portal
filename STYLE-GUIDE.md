# Style guide

This style guide keeps docs consistent and easy to scan.

## Voice and tone

- Clear, direct, and developer-focused
- Avoid marketing language
- Prefer active voice

## Formatting

- Use sentence case headings (example: `# Rate limits`)
- Use fenced code blocks with language tags (`bash`, `json`, `yaml`)
- Use tables for parameters and fields

## API terminology

- “API key” for the Bearer token
- “sandbox” and “production” environments
- Use “idempotency key” for `Idempotency-Key`

## Linking

- Prefer relative links within the repo
- Keep link text meaningful (avoid “click here”)

## Examples

- Provide at least one `curl` example per endpoint
- Provide JSON request and response examples where applicable
